# Requirements Document

## Introduction

This document specifies the requirements for an MVP autonomous **Generative AI + Agentic AI** powered supply chain optimization platform designed for a large Energy and Services company operating across the UK and Republic of Ireland. The system employs **AWS Bedrock Agents** with **Claude 3.5 Sonnet** foundation model to automate procurement decisioning, inventory rebalancing, and demand forecasting while maintaining human oversight for high-risk actions. The platform serves Procurement Managers and Inventory Managers, processing operations for 3 warehouses with 2,000 SKUs and 500 suppliers. The MVP is deployed in **us-east-1 (N. Virginia)** using Streamlit on SageMaker, **Redshift Serverless** for data warehousing, Glue for ETL, Lambda for agent tools, Bedrock for AI reasoning, and IAM for access control, all managed via AWS Console.

## Glossary

- **Platform**: The complete supply chain optimization system powered by Generative AI
- **Bedrock**: AWS managed service for foundation models (Claude 3.5 Sonnet)
- **Bedrock_Agent**: AWS Bedrock Agent with tool-calling capabilities for autonomous decision-making
- **Procurement_Agent**: Bedrock Agent responsible for purchase order creation and supplier selection
- **Inventory_Agent**: Bedrock Agent responsible for stock rebalancing across warehouses
- **Forecasting_Agent**: Bedrock Agent responsible for demand prediction
- **Claude_3.5_Sonnet**: Foundation model from Anthropic used for reasoning and natural language generation
- **Action_Group**: Set of Lambda tools that Bedrock Agents can invoke
- **Procurement_Manager**: User who oversees purchasing decisions and supplier relationships
- **Inventory_Manager**: User who oversees stock levels and rebalancing
- **SKU**: Stock Keeping Unit, a unique product identifier
- **MAPE**: Mean Absolute Percentage Error, forecast accuracy metric
- **Human_Approval_Queue**: System component for routing high-risk decisions to human reviewers
- **Audit_Log**: Log of all agent decisions and actions stored in Redshift Serverless
- **Confidence_Score**: Numerical value (0-1) indicating agent certainty in a decision, generated by LLM
- **Glue_Job**: AWS Glue ETL job for data transformation
- **Lambda_Tool**: AWS Lambda function that serves as a tool for Bedrock Agents
- **SageMaker_Notebook**: Streamlit application hosted on SageMaker
- **Redshift_Serverless**: Serverless data warehouse storing all operational data

## Requirements

### Requirement 1: Autonomous Procurement Agent (Bedrock-Powered)

**User Story:** As a Procurement Manager, I want a Generative AI agent powered by AWS Bedrock to automatically create purchase orders based on demand forecasts and inventory levels, so that I can reduce manual workload while maintaining optimal stock levels with explainable AI reasoning.

#### Acceptance Criteria

1. WHEN inventory levels fall below reorder point for any SKU, THE Bedrock Procurement_Agent SHALL generate a purchase order recommendation with supplier selection using Claude 3.5 Sonnet reasoning
2. WHEN generating a purchase order, THE Bedrock Procurement_Agent SHALL include LLM-generated natural language rationale explaining the decision
3. WHEN generating a purchase order, THE Bedrock Procurement_Agent SHALL calculate and include a confidence score between 0 and 1 based on LLM reasoning
4. IF the purchase order value exceeds £10,000 OR confidence score is below 0.7, THEN THE Bedrock Procurement_Agent SHALL route the decision to the Human_Approval_Queue
5. WHEN a purchase order is approved, THE Bedrock Procurement_Agent SHALL invoke Lambda tools to create a Purchase_Order record in Redshift Serverless with header and line items
6. THE Bedrock Procurement_Agent SHALL select suppliers based on price, lead time, and historical performance using LLM reasoning with tool-provided data
7. WHEN creating purchase orders, THE Bedrock Procurement_Agent SHALL record all decisions to the Audit_Log in Redshift Serverless with timestamp and LLM-generated rationale

### Requirement 2: Autonomous Inventory Rebalancing Agent (Bedrock-Powered)

**User Story:** As an Inventory Manager, I want a Generative AI agent powered by AWS Bedrock to automatically rebalance stock across warehouses, so that inventory is optimally distributed to meet regional demand with AI-driven insights.

#### Acceptance Criteria

1. WHEN inventory imbalance is detected across warehouses, THE Bedrock Inventory_Agent SHALL generate stock transfer recommendations using Claude 3.5 Sonnet reasoning
2. WHEN generating transfer recommendations, THE Bedrock Inventory_Agent SHALL include LLM-generated natural language rationale for each transfer
3. WHEN generating transfer recommendations, THE Bedrock Inventory_Agent SHALL calculate and include a confidence score between 0 and 1 using LLM reasoning
4. IF a transfer involves more than 100 units OR confidence score is below 0.75, THEN THE Bedrock Inventory_Agent SHALL route the decision to the Human_Approval_Queue
5. WHEN a transfer is approved, THE Bedrock Inventory_Agent SHALL invoke Lambda tools to update inventory records in Redshift Serverless for source and destination warehouses
6. THE Bedrock Inventory_Agent SHALL consider regional demand patterns and warehouse capacity when recommending transfers using LLM reasoning with tool-provided data
7. WHEN executing transfers, THE Bedrock Inventory_Agent SHALL record all decisions to the Audit_Log in Redshift Serverless with timestamp and LLM-generated rationale

### Requirement 3: Demand Forecasting Agent (Bedrock-Powered)

**User Story:** As an Inventory Manager, I want accurate demand forecasts for all SKUs generated by AI, so that I can maintain optimal inventory levels and avoid stockouts with intelligent predictions.

#### Acceptance Criteria

1. THE Bedrock Forecasting_Agent SHALL generate demand forecasts for all 2,000 SKUs on a daily basis using Claude 3.5 Sonnet combined with statistical models
2. THE Bedrock Forecasting_Agent SHALL achieve Mean Absolute Percentage Error (MAPE) below 15% for the top 200 SKUs by volume
3. WHEN generating forecasts, THE Bedrock Forecasting_Agent SHALL consider historical sales data and seasonality using LLM reasoning and statistical tools
4. THE Bedrock Forecasting_Agent SHALL provide forecast confidence intervals at 80% and 95% levels
5. THE Bedrock Forecasting_Agent SHALL support forecast horizons of 7 days and 30 days
6. WHEN generating forecasts, THE Bedrock Forecasting_Agent SHALL invoke Lambda tools to store forecast results in Redshift Serverless
7. THE Bedrock Forecasting_Agent SHALL execute as a Bedrock Agent triggered daily by EventBridge

### Requirement 4: Human-in-the-Loop Approval System

**User Story:** As a Procurement Manager, I want to review and approve high-risk AI decisions, so that I maintain control over critical business operations.

#### Acceptance Criteria

1. WHEN an agent decision is routed to the Human_Approval_Queue, THE Platform SHALL display the request in the Streamlit dashboard
2. WHEN displaying approval requests, THE Platform SHALL show the agent's rationale, confidence score, and all relevant data
3. WHEN a user approves a decision, THE Platform SHALL execute the action via Lambda_Function and record the approval to the Audit_Log
4. WHEN a user rejects a decision, THE Platform SHALL cancel the action and record the rejection with user-provided reason to the Audit_Log
5. THE Platform SHALL allow users to modify agent recommendations before approval
6. THE Platform SHALL provide a dashboard view of all pending approvals filtered by user role
7. THE Platform SHALL store approval queue data in Redshift

### Requirement 5: Audit Logging

**User Story:** As a Compliance Officer, I want a complete audit log of all agent decisions and actions, so that I can ensure regulatory compliance and investigate issues.

#### Acceptance Criteria

1. THE Platform SHALL record every agent decision to the Audit_Log table in Redshift with timestamp, agent identifier, decision type, and rationale
2. THE Platform SHALL record every human approval or rejection to the Audit_Log with timestamp, user identifier, and reason
3. THE Platform SHALL record every data modification to the Audit_Log with before and after states
4. THE Platform SHALL provide search and filter capabilities for audit records by date range, agent, and user in the Streamlit dashboard
5. THE Platform SHALL support audit log export in CSV format
6. THE Platform SHALL retain audit records for minimum 7 years in Redshift
7. THE Audit_Log table SHALL include columns for event_id, timestamp, agent_name, user_name, action_type, rationale, confidence_score, and metadata_json

### Requirement 6: AWS Glue Data Ingestion

**User Story:** As a Data Engineer, I want to ingest data from S3 using AWS Glue, so that I can load operational data into Redshift.

#### Acceptance Criteria

1. THE Platform SHALL use AWS Glue_Job to extract data from S3 buckets
2. THE Glue_Job SHALL transform data to match the Redshift schema
3. THE Glue_Job SHALL load transformed data into Redshift tables
4. THE Glue_Job SHALL run on a daily schedule for batch processing
5. WHEN data validation fails, THE Glue_Job SHALL log errors to CloudWatch and skip invalid records
6. THE Glue_Job SHALL track ingestion metrics including record count and success rate
7. THE Platform SHALL support manual trigger of Glue_Job via AWS Console

### Requirement 7: Streamlit Dashboard on SageMaker with Bedrock Integration

**User Story:** As a Procurement Manager, I want a web dashboard with AI chat capabilities to view agent recommendations, interact with Bedrock agents, and approve decisions, so that I can manage supply chain operations efficiently with AI assistance.

#### Acceptance Criteria

1. THE Platform SHALL host the Streamlit application on Amazon SageMaker_Notebook in us-east-1
2. WHEN a user logs in, THE Platform SHALL display a role-specific dashboard based on user type (Procurement_Manager or Inventory_Manager)
3. WHEN a Procurement_Manager views the dashboard, THE Platform SHALL display pending approvals, recent purchase orders, supplier performance, and an AI chat interface for interacting with Bedrock agents
4. WHEN an Inventory_Manager views the dashboard, THE Platform SHALL display inventory levels, stockout alerts, forecast accuracy, rebalancing recommendations, and an AI chat interface
5. THE Platform SHALL implement a glassy theme with modern UI components
6. THE Platform SHALL query Redshift_Serverless for all dashboard data using the Redshift Data API
7. THE Platform SHALL provide interactive filters for date range, warehouse, and SKU, plus natural language query capabilities via Bedrock

### Requirement 8: AWS Bedrock Agent Execution

**User Story:** As a System Architect, I want agents to run as Bedrock Agents with Lambda tools, so that the platform leverages foundation models for intelligent decision-making and scales automatically without managing servers.

#### Acceptance Criteria

1. THE Procurement_Agent SHALL execute as a Bedrock_Agent triggered by EventBridge schedule using Claude 3.5 Sonnet
2. THE Inventory_Agent SHALL execute as a Bedrock_Agent triggered by EventBridge schedule using Claude 3.5 Sonnet
3. THE Forecasting_Agent SHALL execute as a Bedrock_Agent triggered daily using Claude 3.5 Sonnet
4. THE Bedrock_Agents SHALL invoke Lambda_Tools to read and write data to Redshift_Serverless
5. THE Bedrock_Agents SHALL use IAM roles for authentication to AWS services
6. THE Bedrock_Agents SHALL log execution details and LLM reasoning to CloudWatch
7. THE Bedrock_Agents SHALL have timeout configured to 15 minutes maximum per invocation

### Requirement 9: IAM Access Control

**User Story:** As a Security Officer, I want role-based access control using AWS IAM, so that only authorized users can access the platform.

#### Acceptance Criteria

1. THE Platform SHALL use AWS IAM for authentication and authorization
2. THE Platform SHALL define IAM roles for Procurement_Manager and Inventory_Manager
3. WHEN a user attempts to access the Streamlit dashboard, THE Platform SHALL verify IAM credentials
4. THE Platform SHALL use IAM roles for Lambda_Function execution with least privilege permissions
5. THE Platform SHALL use IAM policies to control access to Redshift_Cluster
6. THE Platform SHALL use IAM policies to control access to S3 buckets
7. THE Platform SHALL log all authentication attempts to CloudWatch

### Requirement 10: Redshift Serverless Data Warehouse

**User Story:** As a Data Engineer, I want all operational data stored in Redshift Serverless, so that I can run analytics queries efficiently without managing infrastructure.

#### Acceptance Criteria

1. THE Platform SHALL store all operational data in Redshift_Serverless
2. THE Redshift_Serverless SHALL include tables for Product, Warehouse, Purchase_Order, Sales_Order, Supplier, Inventory, and Audit_Log
3. THE Redshift_Serverless SHALL be deployed in us-east-1 (N. Virginia) for optimal Bedrock integration
4. THE Redshift_Serverless SHALL use 32 RPUs base capacity for MVP with auto-scaling enabled
5. THE Redshift_Serverless SHALL have automated snapshots enabled with 7-day retention
6. THE Platform SHALL use Redshift Data API for serverless connectivity from Lambda tools and Streamlit
7. THE Platform SHALL manage Redshift_Serverless via AWS Console only

### Requirement 11: Synthetic Data Generation

**User Story:** As a Developer, I want synthetic operational data for testing, so that I can develop and test the platform without accessing production data.

#### Acceptance Criteria

1. THE Platform SHALL generate 12 months of synthetic sales order data
2. THE Platform SHALL generate synthetic product data for 2,000 SKUs with realistic attributes
3. THE Platform SHALL generate synthetic supplier data for 500 suppliers with performance metrics
4. THE Platform SHALL generate synthetic warehouse data for 3 warehouses (WH1_South, WH_Midland, WH_North)
5. THE Platform SHALL generate synthetic purchase order data with realistic lead times and quantities
6. THE Platform SHALL ensure synthetic data follows realistic patterns including seasonality
7. THE Platform SHALL provide Python scripts to generate and load synthetic data into Redshift

### Requirement 12: Agent Explainability (LLM-Powered)

**User Story:** As a Procurement Manager, I want to understand why AI agents make specific decisions using natural language explanations from the LLM, so that I can trust the system and learn from its recommendations.

#### Acceptance Criteria

1. WHEN an agent makes a decision, THE Platform SHALL use Claude 3.5 Sonnet to generate a natural language explanation of the reasoning
2. THE Platform SHALL display the top 3 factors that influenced each agent decision with relative importance weights, as determined by the LLM
3. THE Platform SHALL store LLM-generated explanations in the Audit_Log table in Redshift_Serverless
4. THE Platform SHALL display LLM-generated explanations in the Streamlit dashboard alongside recommendations
5. THE Platform SHALL include confidence scores in all explanations, calculated by the LLM
6. THE Platform SHALL allow users to ask follow-up questions to Bedrock agents about historical decisions via natural language chat
7. THE Platform SHALL track agent decision accuracy over time and use this data to improve LLM prompts

### Requirement 13: Inventory Optimization Metrics

**User Story:** As an Inventory Manager, I want to track inventory performance metrics, so that I can measure the impact of AI-driven optimization.

#### Acceptance Criteria

1. THE Platform SHALL calculate and display inventory turnover ratio for each warehouse
2. THE Platform SHALL track stockout incidents and calculate stockout rate by SKU
3. THE Platform SHALL measure the improvement in inventory turns compared to a baseline period
4. THE Platform SHALL achieve minimum 10% improvement in inventory turns over 6-month baseline
5. THE Platform SHALL track slow-moving SKUs and flag them in the dashboard
6. THE Platform SHALL provide trend charts showing inventory metrics over time
7. THE Platform SHALL store all metrics in Redshift for historical analysis

### Requirement 14: Supplier Performance Tracking

**User Story:** As a Procurement Manager, I want to track supplier performance, so that I can make informed decisions about supplier selection.

#### Acceptance Criteria

1. THE Platform SHALL calculate supplier reliability score based on on-time delivery rate
2. THE Platform SHALL track average lead time by supplier
3. THE Platform SHALL track supplier defect rates
4. THE Platform SHALL provide supplier scorecards in the Streamlit dashboard
5. THE Platform SHALL alert Procurement Managers when supplier performance degrades below thresholds
6. THE Platform SHALL use supplier performance data as input to the Procurement_Agent's supplier selection logic
7. THE Platform SHALL store supplier metrics in Redshift
